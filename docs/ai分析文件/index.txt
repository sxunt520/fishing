**所有函数 / 方法（签名、参数、返回、行为简述）**
- `fetchSpots = async (r: Region) => { ... }`  
  - 参数：`r: Region`  
  - 返回：Promise<void>（函数没有显式返回值）  
  - 行为：调用 `spotApi.getInBounds({ north, south, east, west })`，计算边界为 `r.latitude ± r.latitudeDelta/2` 与 `r.longitude ± r.longitudeDelta/2`，然后 `setSpots(res || [])`；错误时 `console.error(e)`。  
  - 代码片段：`const res: any = await spotApi.getInBounds({ north: r.latitude + r.latitudeDelta / 2, ... }); setSpots(res || []);`

- `onRegionChangeComplete = (r: Region) => { setRegion(r); fetchSpots(r); }`  
  - 参数：`r: Region`  
  - 返回：`void`  
  - 行为：更新状态并拉取新的钓点。

- `zoom = (factor: number) => setRegion((p) => ({ ...p, latitudeDelta: p.latitudeDelta * factor, longitudeDelta: p.longitudeDelta * factor, }));`  
  - 参数：`factor: number`  
  - 返回：`void`（状态更新）  
  - 行为：按系数缩放 delta（放大传入 `0.5`，缩小 `2` 等）。

- `goToMyLocation = async () => { if (!currentLocation) { await requestLocation(); return; } mapRef.current?.animateToRegion({ latitude: currentLocation.latitude, longitude: currentLocation.longitude, latitudeDelta: 0.01, longitudeDelta: 0.01, }); }`  
  - 参数：无  
  - 返回：Promise<void>  
  - 行为：若 `currentLocation` 不存在，则调用 `requestLocation()` 并直接返回（不会在获取后继续移动）；若存在则调用 `mapRef.current?.animateToRegion(...)`。  
  - 注意：`requestLocation()` 的结果并未在该函数内继续处理或等待后动画。

- `handleMarkerPress = useCallback((spot: any) => { setSelectedSpot(spot); sheetRef.current?.snapToIndex(0); }, []);`  
  - 参数：`spot: any`  
  - 返回：`void`  
  - 行为：设置选中钓点并将 `BottomSheet` 弹到索引 0（顶部 snap point）。使用空依赖数组创建固定回调。

**潜在问题、边界情况与不常见模式**
1. useEffect 依赖问题：  
   - `useEffect(..., [location])` 内部调用了 `fetchSpots`，但 `fetchSpots` 没有列入依赖。虽然 `fetchSpots` 在函数组件范围内且不会变动（每次渲染都会重新创建），这会让 ESLint 的 hooks 规则报错。建议将 `fetchSpots` 用 `useCallback` 包装或在 `useEffect` 内直接内联实现请求逻辑。  
   - 代码片段："useEffect(... fetchSpots(r) ... , [location]);"

2. `goToMyLocation` 逻辑不完整：  
   - 当 `currentLocation` 为 `undefined` 时，函数 `await requestLocation(); return;` —— 在请求到位置后不会自动调用 `animateToRegion`，需要额外处理（比如等待定位完成或监听 store 的 `currentLocation` 变化再移动）。代码片段：`if (!currentLocation) { await requestLocation(); return; }`

3. `fetchSpots` 对 API 返回的假设：  
   - `const res: any = await spotApi.getInBounds(...); setSpots(res || []);` 假设 `res` 就是数组；若 API 返回 `{ data: [...] }` 或类似格式，这会出问题。应该明确 API 返回类型并处理错误响应。  

4. 类型安全不足：多处使用 `any`（`spots`, `selectedSpot`, `icon`）。这会掩盖属性访问或类型错误（例如 `selectedSpot?.id` 假设存在 `id` 字段）。  

5. 并发/节流问题：`onRegionChangeComplete` 每次区域结束变化都会触发 `fetchSpots`，若用户频繁移动地图可能会造成多次请求。建议节流（debounce）或只在一定 delta 变化/停止时请求。  

6. 错误处理与用户反馈：`fetchSpots` 的 `catch` 仅 `console.error(e)`，没有 UI 层反馈或重试策略。`BottomSheet` 打开错误也未处理。  

7. `handleMarkerPress` 的 `useCallback` 依赖数组为空（`[]`），但它使用 `sheetRef` 和 `setSelectedSpot`（后者稳定），`sheetRef` 为 ref 不会改变，通常可接受，但 ESLint 可能仍警告。  

8. `MapSurface` 与 `MapSurfaceHandle` 假设：组件需正确实现 `forwardRef` 并提供 `animateToRegion` 方法，否则 `mapRef.current?.animateToRegion` 可能是 undefined。

9. Accessibility / a11y 缺失：`TouchableOpacity` 未添加 `accessibilityLabel` / `accessibilityRole`。

**小改进建议（优先级由高到低）**
- 类型化：定义 `type Spot = { id: string; latitude: number; longitude: number; ... }`，替换 `any`。将 `ToolButton` 的 `icon` 类型改为 `keyof typeof Ionicons.glyphMap` 或 `string` 更精确。  
- 修正 useEffect / fetchSpots 依赖：把 `fetchSpots` 用 `useCallback` 包装，或在 `useEffect` 内写入请求逻辑，确保 ESLint hooks 规则通过。  
- 完善 `goToMyLocation`：在 `!currentLocation` 的分支，等待定位后再次读取 `currentLocation` 并调用 `animateToRegion`，或在 `useEffect` 监听 `currentLocation` 的变化来动画跳转。  
- 防抖/节流 `fetchSpots`：对 `onRegionChangeComplete` 的调用做 debounce（例如 300-500ms），避免抖动导致大量请求。  
- 明确处理 API 返回结构：检查 `spotApi.getInBounds` 的返回类型并正确解析（例如 `res.data`）。  
- 更好的错误处理与加载状态：在拉取时显示 loading 指示器、空状态处理和错误提示。  
- 增加 `accessibilityLabel` 到可点击元素，改善无障碍支持。  
- 在 `BottomSheet` 使用 `selectedSpot` 为 `null` 的情况显示占位或隐藏组件，以避免内部组件对 `undefined` id 的崩溃。  
- 考虑将初始 `region` 从硬编码北京坐标改为更合理的默认（或只在无定位权限时使用）。
